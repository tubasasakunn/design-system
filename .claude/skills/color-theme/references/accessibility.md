# 色のコントラストとアクセシビリティ リファレンス

> color-theme スキルの参考資料。閾値・計算式は 2026-06 時点の現行仕様。

## 0. 現行仕様のステータス（前提）

- **WCAG 2.2** が現行のW3C勧告（2023-10-05、更新2024-12-12）。**ISO/IEC 40500:2025** として国際標準化。
- WCAG 2.0/2.1/2.2 は後方互換。コントラスト中核基準（1.4.1/1.4.3/1.4.6/1.4.11）は**閾値も式も2.0〜2.2で不変**。
- 各国法規（米 ADA/Section 508、欧 EN 301 549、日 JIS X 8341-3:2016≒WCAG 2.0）が参照するのは **WCAG 2 系**。**実務・法的に守るべきは WCAG 2（多くは AA）**。
- **WCAG 3.0 はまだ草案**。視覚コントラスト章（APCAを含む）は本体から外れたまま。**現時点で APCA は規範ではない**。最終勧告は早くて2028年以降、現実的に2030年前後。

## 1. WCAG 2.x のコントラスト要件

### 1.1 コントラスト比の計算式

**相対輝度(relative luminance) L** から計算する。

1. **正規化**: 8bit値を255で割り `Rs,Gs,Bs`（0–1）。
2. **ガンマ展開（線形化）** 各 `c`:
   ```
   clin = c/12.92                    (c ≤ 0.04045)
   clin = ((c+0.055)/1.055)^2.4      (c > 0.04045)
   ```
3. **相対輝度**: `L = 0.2126·Rlin + 0.7152·Glin + 0.0722·Blin`（緑が支配的、青の寄与小。黒=0, 白=1）
4. **コントラスト比**: `(L1 + 0.05) / (L2 + 0.05)`（L1=明るい方, L2=暗い方）
   - 範囲 **1:1 〜 21:1**（純黒×純白）。`+0.05`は環境光(flare)の近似。暗部の微差が比に効きにくい原因。

### 1.2 SC 1.4.3 Contrast (Minimum) — AA

| 区分 | 最小比 |
|---|---|
| **通常テキスト** | **4.5:1** |
| **大きいテキスト** | **3:1** |

**「大きいテキスト」の定義**: 18pt以上、または14pt以上かつ太字(bold)。CSS換算 **1pt = 1.333…px(4/3)** より **18pt≈24px**、**14pt bold≈18.66px**。bold≈font-weight 700以上。
**例外**: 装飾テキスト、ロゴ/ブランド名、**無効化(disabled)コンポーネントのテキスト**。

### 1.3 SC 1.4.6 Contrast (Enhanced) — AAA
通常 **7:1** / 大きいテキスト **4.5:1**。長文本文・医療/金融/公共・高齢者向けで目標にする価値あり。

### 1.4 SC 1.4.11 Non-text Contrast — AA（2.1で追加）
テキスト**以外**は隣接色に対し **3:1 以上**。
- **UIコンポーネント**: ボタン/入力欄の境界、チェック/ラジオの枠、トグルのつまみ、**フォーカスインジケータ**、状態を示す装飾。
- **グラフィカルオブジェクト**: 情報を伝えるアイコン、折れ線/円グラフの要素。
- **例外**: 無効化コンポーネント、色が意味そのもの（写真・ヒートマップ・国旗・地図色分け）、ブラウザ既定描画、純装飾。

> アイコンが `stroke="currentColor"` でテキスト色を継承する設計なら、**テキストが4.5:1を満たせば情報アイコンの3:1も自動充足**。ただし淡い面・フォーカスリング・disabledは個別確認。

### 1.5 早見表

| 基準 | レベル | 対象 | 閾値 |
|---|---|---|---|
| 1.4.3 | AA | 通常テキスト | 4.5:1 |
| 1.4.3 | AA | 大きいテキスト(≥24px / ≥18.66px bold) | 3:1 |
| 1.4.6 | AAA | 通常 / 大きい | 7:1 / 4.5:1 |
| 1.4.11 | AA | UIコンポーネント・グラフィック | 3:1 |

## 2. WCAG 2 式の既知の弱点

1. **暗い色域で不正確**: `+0.05` フレア項が暗部で支配的になり比が圧縮される。濃色背景上の濃色テキストは数値と体感がズレる（両方向に）。
2. **ダークモードで体感乖離**: 同じ比でも**正極性(明背景に暗文字)**と**負極性(暗背景に明文字)**で知覚が違う（暗背景・明文字はハレーションしやすい）。WCAG 2 は**極性を区別しない**。
3. **輝度のみで色相・彩度を無視**: 輝度が近い赤と緑を機械的に弾く一方、知覚的に区別できる配色も弾くことがある。
4. **空間特性（太さ・サイズ）を式に織り込めない**（大/小の2段階のみ）。

→ これらの問題意識から知覚ベースの **APCA** が生まれた。

## 3. APCA と WCAG 3.0 の現状

### 3.1 APCA とは
知覚ベースのコントラスト評価。出力は **Lc（Lightness Contrast）値**（比ではなくスコア、概ね 0〜±106）。
- **極性を考慮**: **正Lc=明背景に暗文字**、**負Lc=暗背景に明文字**。符号が「どちらが背景か」を表す。
- **均等知覚に近い**: Lc 60 はパステルでも濃色でもほぼ同じ知覚コントラスト。
- **フォントのサイズ・太さとセットで閾値**を決める。

### 3.2 Lc 閾値（目安、参照フォント Arial 400）

| Lc | 位置づけ |
|---|---|
| **90** | Preferred body（本文推奨。14px/400可） |
| **75** | Minimum body（本文最低。16px/400程度） |
| **60** | Minimum fluent（読みやすい一般テキスト最低。24px/400程度） |
| **45** | Minimum large（大文字最低。42px/400程度） |
| **30** | 非本文・ラベルのみ |
| **15** | 実質視認不能 |

> 大づかみ: **WCAG 4.5:1 ≒ Lc 60前後**、**7:1 ≒ Lc 75〜90前後**（色域で大きくずれる近似）。本文は **Lc 75以上**が堅い。

### 3.3 WCAG 3.0 ステータス（2026-06）
Working/Editor's Draft 段階。コントラストアルゴリズムは **未定(yet to be determined)**。**APCA は規範として含まれていない**。**法的・規範基準は WCAG 2.2（AA が事実上の標準）**。

### 3.4 実務での併用方針
1. **合否判定の正本は WCAG 2（AA、必要ならAAA）**。監査・契約・法対応はこれ。
2. **APCA は設計の質を上げる補助**: 特に**ダークモードの文字色チューニング**（極性を見られる）、**淡いアクセント面の文字**や**濃色面の濃色テキスト**（WCAG 2の暗部弱点が出る箇所）の検証。
3. APCA採用でも**最終的に WCAG 2 も満たすか確認**。外れる場合は不適合を文書化。
4. トークンに **「WCAG 2比」と「APCA Lc」を両方記録**しておくと将来移行が容易。

## 4. 色覚多様性（Color Vision Deficiency, CVD）

| 型 | 通称 | 内容 | 頻度 |
|---|---|---|---|
| **1型 protan** | 赤系 | L錐体(赤)の欠損/異常 | 男性 約2% |
| **2型 deutan** | 緑系 | M錐体(緑)。**deuteranomaly が最多 男性約5%** | 男性 約6% |
| **3型 tritan** | 青黄系 | S錐体(青)。非常にまれ | ごくまれ・性差ほぼなし |

- **赤緑系(1型+2型)合計: 男性 約8%、女性 約0.5%**（北欧系は男性10–11%）。1型・2型はX連鎖で男性に多い。
- 数十〜百人規模で CVD ユーザーがほぼ確実に含まれる前提で設計。

**UI配慮**:
- **赤×緑だけで成否/正誤を区別しない**（最多）。青×紫・緑×茶・暗赤×黒も紛らわしい。
- **輝度差(明暗差)を必ずつける**（色相が判別できなくても明暗で区別可。WCAG比を満たす配色はCVDにも有利）。
- データ可視化は色数を絞り、明度を変える/パターン/直接ラベル。**CVDセーフパレット**（ColorBrewer CVD-safe、Okabe-Ito 8色）を採用すると安全。

**シミュレーションツール**: Chrome DevTools の Rendering → Emulate vision deficiencies、Color Oracle、Coblis、Sim Daltonism、Figma の Stark。

## 5. 「色だけに依存しない」原則（SC 1.4.1 — レベル A）

**色を、情報伝達・操作指示・区別の「唯一の」手段にしない。** 最も基本的な必須レベル。

| 場面 | NG（色のみ） | OK（冗長化） |
|---|---|---|
| エラー | 枠を赤くするだけ | 赤枠 ＋ アイコン ＋ テキスト |
| 必須項目 | 赤文字のみ | 赤文字 ＋「*」/「必須」 |
| ステータス | 緑/黄/赤の点のみ | 色 ＋ 形(✓/!/×) ＋ ラベル |
| リンク | 色だけで本文と区別 | **下線**（ホバー以外でも識別可能に） |
| グラフ | 線/塗りの色だけ | 線種・マーカー形状・パターン・直接ラベル |

- **status トークンは必ずアイコン/テキストとセット**で使う運用ルールに。
- 1.4.1 は「色を使うな」ではなく「**色以外の手がかりも併せ持て**」という冗長性の要求。

## 6. ダーク/ライト両モードでコントラストを担保する実務

### 6.1 純黒・純白を避けつつ確保
- 本文に `#000`/`#fff` を多用しない。暗背景の純白はハレーション、明背景の純黒は強すぎて疲労。
- **オフブラック(#111〜#1a1a1a系) / オフホワイト(#f5f5f5〜#fafafa系)** を使い、**それでも本文4.5:1を満たす**。コントラストは「最大化」でなく「十分(≥4.5:1)かつ快適」を狙う。
- ダーク背景はわずかに彩度を持たせた暗グレー/暗ネイビーにすると階層を表現しやすい。

### 6.2 淡いアクセント面の上のテキスト
- 薄い色の通知バー/バッジ/カードでは、**面の色ではなく面の上に乗るテキスト色**を背景(淡い面)に対し4.5:1で測る。薄緑面には**濃い緑文字**を。
- ダークでアクセント面を作るなら明るい原色そのままは不可。**彩度を落とし暗くしたトーン**にし、文字は明るいアクセント or オフホワイトで4.5:1確保。

### 6.3 ダーク用に色を「作り直す」
- **ライト値を反転/流用しない**（`light.json`/`dark.json` を別最適化するのは正しい）。
- ダークは**彩度を少し下げ明度を上げる**と読みやすい。**アクセントはダークで明るめ**に（暗背景に3:1〜4.5:1）。
- status は色相を維持しつつ明度を各モードで調整。

### 6.4 トークン設計上の指針
- 各ペアでライト/ダーク両方の目標コントラストを定義:
  - 本文 `fg.primary × bg.primary`: ≥4.5:1（できれば7:1）。
  - 補助テキスト/プレースホルダ(`fg.secondary`/`fg.muted`): **3:1で妥協せず最低4.5:1**を狙う。「グレーすぎる二次テキスト」は典型的不合格箇所。
  - 境界・アイコン・フォーカス(`border.*`/`accent.*`): ≥3:1。
- キー集合をライト/ダークで揃え、**ビルド時/CIでコントラストを自動検証**すると回帰を防げる（このスキルの `scripts/check-contrast.mjs` 参照）。

## 7. 実務チェック手順とツール

1. **設計段階**: パレット候補を contrast checker / Figma プラグインで AA 確認。CVDシミュレータで赤緑・青黄確認。
2. **トークン定義時**: 主要 `fg×bg`・状態色・境界/フォーカスの WCAG 2 比を算出記録（可能ならAPCA Lcも）。
3. **実装段階**: axe-core / Lighthouse / Pa11y で 1.4.3・1.4.11 違反をCI検出。
4. **手動確認**: テキスト on 画像/グラデ、フォーカスリング、disabled除外、グラフ系列を目視。
5. **CVD・両モード確認**: ライト/ダーク両方で protan/deutan/tritan を通す。
6. **1.4.1監査**: 色のみで意味を持たせた箇所(エラー・必須・グラフ・リンク)を点検。

**ツール**: WebAIM Contrast Checker、TPGi Colour Contrast Analyser(CCA)、Chrome DevTools（比＋AA/AAAライン）、APCA Calculator(apcacontrast.com)、axe/Lighthouse/WAVE、ColorBrewer/Leonardo。
