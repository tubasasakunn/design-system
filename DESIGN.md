# DESIGN.md — デザイン思想と審美眼

このリポジトリは「ひとつの正解のテーマ／アイコン」を作る場所ではない。
**複数のカラーテーマと複数のアイコンスタイル（カクカク系・丸み系・塗り系…）を、
オーナー個人の一貫した審美眼で選び・束ね・一元管理するための場所**である。

`CLAUDE.md` が *仕組み*（single source of truth とマルチプラットフォーム生成）を定義するのに対し、
この `DESIGN.md` は *審美眼* を定義する。つまり「何を採用し、何を採用しないか」「複数を持つときどう整理するか」の判断基準である。
新しいテーマ・アイコンを増やすたびに、ここに照らして「これは“自分の好み”の集合に入るか」を判定する。

判断の根拠は、好み診断ツール（`/tmp/design-preference/`）の回答（24問・対照比較）と、オーナーの定性コメント。
スコアの確度（confidence）も併記しているので、強い軸は厳守、弱い軸は余地あり、として扱う。

---

## 0. 大原則 — 「複数を持つ」が前提

- **カラーテーマは複数持つ。** それぞれが light / dark の外観を内包する（`CLAUDE.md` のルール通り、同一テーマの light/dark は同じトークンキー集合）。
- **アイコンは複数スタイル系統を持つ。** カクカク系（sharp）と丸み系（rounded）を別物として併存させ、塗り系なども将来追加しうる。
- だが **すべては同じ審美眼（§2 のハウススタイル）を通過したものだけ** を採用する。多様性は「無秩序」ではなく「好みの幅」として持つ。
- 各カテゴリには **フラッグシップ（既定）** を1つ決める。迷ったらこれ、という基準点。
  - 既定テーマ … `cream`（生成り × パステルティール、ライト基調）
  - 既定アイコン … `rounded`（丸み系・線画・2px）

## 1. 審美眼プロファイル（診断の要約）

| 軸 | 結論 | 確度 | 備考 |
|---|---|---|---|
| 既定の外観 | **ライト**基調 | ★★★ 高 | 最も強い信号。ダークも整備するが主役はライト |
| ベースの色温度 | **ウォーム**（生成り・クリーム） | ★★ 中 | 目にやさしい温かい白を好む |
| 彩度 | **パステル〜低彩度** | ★★ 中 | 鮮やかなベタ塗りは明確に却下 |
| アクセント色相 | **ティール**（断トツ）／次点レッド | ★★ 中 | ティールが「相棒の色」 |
| アクセントの濃さ | **パステル基調**（淡いティール主役＋濃色文字） | — | 可読性はコントラスト管理で担保 |
| 差し色 | **パステル多色**（teal 軸＋pink/yellow/lavender…用途で） | — | 図解・カテゴリ分け向き |
| アイコンの線 | **線画**・中〜やや細（**2px**）・**角丸** | ★★ 中 | 塗りは支配的でない |
| アイコンの密度 | **ミニマル**・余白多め | ★ 弱 | 削ぎ落とした記号性 |
| UI の角丸 | **しっかり丸い（~20px）** | ★★ 中 | カード・ボタンとも丸め |
| 区切り線 | **控えめ（ほぼ無し）** | ★★ 中 | 境界は色面と余白で表現 |
| 奥行き | **やわらかい影**（控えめ） | — | フラットでも過剰な影でもない |
| コントラスト | **やわらか〜中** | ★ 弱 | きつい黒文字は避ける |
| 余白の密度 | **標準〜ゆったり** | ★ 弱 | 詰め込まない |
| ステータス色 | **鮮やかでOK** | ★★ 中 | 成功/警告/エラーは視認性優先で例外的に彩度を上げてよい |
| 世界観 | **静かなミニマル × やわらかさ** | ★★ 中 | Loose Drawing 的な“ゆるく親しみ”の温度を許容 |

一言でいえば — **「生成りの紙に、淡いティールと手描きのような線。静かで、やわらかく、余白の効いたミニマル。」**

## 2. ハウススタイル（すべてが通過する不変項）

複数テーマ・複数アイコンを横断して守る「これを外すと“自分の好みじゃない”」という境界。
新規追加時のチェックリストとして使う。

1. **やさしい明度** — 純白の強い白さ・漆黒の沈み込みは避け、生成り〜やわらかい白／温かい暗がりを基調にする。
2. **低〜中彩度をデフォルトに** — 面・背景・アクセントはパステル/くすみ寄り。鮮やかさは「ステータス」「強い CTA」など意味のある一点に限定する。
3. **可読性は妥協しない** — パステルを多用するぶん、文字とのコントラストは別管理（§3.5）。淡い色の上には濃色の文字、淡い色を文字に使わない。
4. **角は丸い** — UI もアイコンも基本は丸み。カクカク系アイコンは「意図したアクセント」として持つが、システムの地は丸み。
5. **境界は引かず、面と余白で分ける** — 強い罫線に頼らない。区切りは背景色差・余白・やわらかい影で。
6. **静けさ＞情報量** — 要素を足すより削る。ミニマルを保ったうえで、手描き的なやわらかさで温度を足す。
7. **ライト優先・ダーク整備** — 主役はライト。ダークは「対応している」状態を常に保つ（トークンキー一致）。

## 3. カラー

### 3.1 思想
紙のような **温かいベース** の上に、**淡いティール** を主役の差し色として置く。
それを補助する **パステル多色**（pink / yellow / lavender / coral …）はカテゴリ分けや図解のための語彙で、
すべて同じ淡いトーンに揃えることで「賑やかだけど騒がしくない」状態を作る。
鮮やかさを許すのはステータス色だけ。

### 3.2 フラッグシップテーマ `cream`（light）

### 3.4 パステル多色パレット（カテゴリ／図解用の語彙）
セマンティックトークンとは別レイヤーの「彩りパレット」。カテゴリ色・タグ・グラフ系列に使う。
各色は **淡い面色＋濃いインク（文字）** のペアで持ち、淡い色を文字色に使わない。

| 名前 | 面 | インク |
|---|---|---|
| teal（主役） | `#5bbcb1` | `#07312d` |
| pink | `#f0a8bd` | `#5b1f31` |
| yellow | `#f1d189` | `#5a4410` |
| lavender | `#bcb2e8` | `#2f2960` |
| coral | `#f3a890` | `#5d2616` |
| sky | `#9fc8ea` | `#143a55` |
| mint | `#a9dcc0` | `#134029` |

> 追加検討トークン: `color.accent.strong`（例 `#2c7d72`）。
> 淡い `accent.default` は cream 地の上では小さな文字／アイコンに対しコントラスト不足。
> **アクセント色のテキスト・線アイコンには濃いめのティールを使う**ため、用途別トークンとして足すのが望ましい
> （`accent.default`=塗り面、`accent.strong`=on-light のテキスト/アイコン、`accent.hover`=操作中）。

### 3.5 コントラスト指針
- 本文テキストは `fg.primary` × `bg.primary` で **4.5:1 以上** を満たす（`#2c2a26`×`#faf7f1` は十分）。
- **パステルの上には必ず濃色インク**（§3.4 のペア／`accent.fg`）。淡い色同士を重ねない。
- きつい純黒（`#000`）は使わず温かい墨（`#2c2a26`）に寄せる＝「やわらかいコントラスト」を保つ。

### 3.6 テーマ・バリエーション（複数持つ前提のラインナップ）
フラッグシップ以外も「好みの幅」として意図的に持つ。各テーマは light/dark を内包し、トークンキーは共通。

| テーマ | 性格 | メモ |
|---|---|---|
| `cream` ★既定 | 生成り × パステルティール | 審美眼の中心 |
| `default`（現行） | 純白 × ブルー | ニュートラルな基準。残すなら「素直な白」枠として |
| `ocean`（現行） | 深いティール/シアンのダーク | 「濃い・テック寄り」の対極バリエーションとして許容 |
| `mist`（候補） | クールなニュートラル白 | 青み白が欲しい日用 |
| `dusk`（候補） | ウォームなダーク主役 | cream の dark をテーマ昇格した版 |

> 新テーマを足すときの判定: **§2 のハウススタイルを通るか**。
> 通らないもの（高彩度ベタ塗り基調・強罫線・純黒沈み込み）は「好みではない」として採用しない。
> ※ ステータス色だけは各テーマで鮮やかさを許容してよい。

## 4. アイコン

### 4.1 複数スタイルを管理する構造（実装済み）
アイコンは「カクカク系」「丸み系」など **スタイル系統が複数併存** する。これを **スタイル軸** として原本構造に組み込んでいる。

**構造:**
```
sources/icons/<style>/<category>/<name>.svg
  例) sources/icons/rounded/navigation/arrow-left.svg
      sources/icons/sharp/navigation/arrow-left.svg   ← 同名 SVG を別スタイルで併存できる
```
- `name` の一意性は **スタイル内で**担保（`rounded` の中で一意／`sharp` の中で一意）。`<style>/<category>/<name>` の三つ組で全体一意。
- 生成側は style を名前空間として扱う:
  - **Web** … `dist/web/icons/<style>/<category>/<name>.svg`、`icons.ts` は `{ rounded: [...], sharp: [...] }` 形＋`IconStyle`/`IconName`/`defaultIconStyle`。
  - **Swift** … `Icons.xcassets` を style フォルダで名前空間化（asset 名 `<style>/<name>`）、API は `DSIcon.Rounded.arrowLeft.image` / `DSIcon.Sharp.arrowLeft.image`。
- **既定スタイルは `rounded`**（`build/lib.mjs` の `defaultIconStyle`。慣例で `rounded`、無ければ先頭スタイル）。
- 新しい style（`hand` など）は `sources/icons/<style>/` を足すだけでビルドが自動で拾う。命名・既定方針を変える判断は ADR（`/adr`）に記録する。

### 4.2 ハウス署名（全スタイル・全アイコン共通の不変トリートメント）
**「借り物のセット」ではなく「自分のセット」にするための統一署名。** 原画の出自（Lucide 等）に関わらず、
**全アイコンに完全に同一の幾何トリートメントを焼き込む**。これが *オリジナリティ*（Lucide 素のままと明確に違う佇まい）と
*統一感*（1203＋アクセントの全点が寸分違わず同じ作法）を両立させる中核。`build` で機械的に一律適用するため逸脱が生じない。

- **ゆったりインセット** … 中心 (12,12) を不動点に **0.875 倍**へ縮小（`<g transform="translate(1.5 1.5) scale(0.875)">` で全子要素を包む）。
  グリフのライブエリアは実効 **~18.4px**（各辺 ~2.8px の余白）。Lucide 素の詰まった作画（実効 ~22px）より **余白が約2.5倍** ＝ 静かでミニマル、空気の通った佇まい（§1「余白多め」「静けさ＞情報量」）。
- **やわらかい光学ストローク** … 属性は `stroke-width="2"` を保ちつつ、上記スケールにより **実効 1.75px**。Lucide の硬い 2px より gentle（§1「やわらかいコントラスト」「やわらかさ」）。
- 不動点が中心なので **対称形は真に中央**のまま（光学センタリングを壊さない）。
- 着色前提（`stroke="currentColor"` / `fill="none"`）。固定色を埋めない。`viewBox="0 0 24 24"`・`width`/`height` 無しは不変。
- 純粋な標準 SVG（`<g transform>` ＋属性のみ。フィルタ・ラスタライズ不使用）なので **Web / Swift 双方で同一にレンダリング**できる（マルチPF の統一感）。

### 4.3 既定スタイル `rounded`（地）
ハウス署名（§4.2）に加え、**端も角も丸い**作法。診断の「丸み系・線画・ミニマル」を体現する“地”。

- `stroke-linecap="round"`、`stroke-linejoin="round"`。
- **ミニマル** — 装飾ディテールを足さず、意味が伝わる最小の線で。

### 4.4 もう一つのスタイル `sharp`（カクカク系・意図したアクセント）
ハウス署名（§4.2 の **同じインセット・同じ実効ストローク**）を共有し、**端と角だけ角ばらせる**ことで丸みの対極を作る。
空気感は共通・終端処理だけ違う → セット横断の統一感を保ったままの差別化。

- `stroke-linecap="butt"`、`stroke-linejoin="miter"`（**端も角も角ばる**）。
- 幾何学的にきっちり。手描き感は出さない。
- 用途: 機能的・テクニカルな文脈、`ocean` のようなテック寄りテーマと相性。

### 4.4 将来スタイルの余地
- `soft-fill`（塗り系・角丸シルエット）… 塗りは支配的でないが、状態表現（選択中など）に部分採用しうる。
- `hand`（手描き・ゆるい系 / Loose Drawing 的）… 「やわらか親しみ」を強めたいテーマ専用に。導入時は線の不揃いさを “味” として許容。

いずれも **既定にはしない**。地は `rounded`、他は意図あるアクセント。

## 5. 形・余白・奥行き（UI の質感）

テーマ非依存の「手触り」。Web の追加トークン or Swift 側の定数として共有する想定。

| 項目 | 値 / 方針 |
|---|---|
| 角丸（カード） | `radius-lg: 20px` を基準。`radius-md: 12px`、`radius-sm: 8px`、`radius-pill: 999px` |
| 角丸（ボタン） | カードと同系で `12–16px`（過度に四角くしない） |
| 境界線 | 既定は **引かない**。必要時のみ `border.default` を 1px。罫線より色面・余白で分離 |
| 奥行き | **やわらかい影** … `0 2px 6px rgba(40,35,25,.08)`。フラットでも強影でもない |
| 余白密度 | **標準〜ゆったり** … カード内 padding 16–20px、要素間 12–16px を基準 |
| コントラスト | やわらか寄り。純黒・純白の最大コントラストは避ける |

## 6. 世界観（トーン＆マナー）

- **静かなミニマル** を地に、**やわらかさ・親しみ** を温度として乗せる。
- 紙のような生成り、淡いティール、丸い角、手描きのような線 — 全体として「やさしくて、少しだけ温かい道具」。
- にぎやかさが要る場面（カテゴリ・図解）は **パステル多色** で、彩度ではなく色相数で楽しさを出す。
- 強さ・緊張感（鮮やかな赤、角ばり、強コントラスト）は **意味のある一点** にだけ使う。

---

## 7. やらないこと（審美眼の境界）

- ❌ 高彩度のベタ塗りを基調にする（鮮やかさはステータス／一点 CTA に限定）
- ❌ 純白の冷たい地・純黒の沈む地を主役にする（ウォーム基調を外す）
- ❌ 強い罫線でグリッドを刻む（面と余白で分ける）
- ❌ 最大コントラストの黒文字（温かい墨に寄せる）
- ❌ 情報を詰め込む高密度レイアウト（静けさ＞情報量）
- ❌ アイコンを1スタイルに強制する／逆にスタイルを無秩序に混在させる（地は `rounded`、他は意図あるアクセント）
- ❌ 淡い色を小さな文字・細い線に使う（可読性を割る）

## 8. 次のアクション

1. **`cream` テーマを実体化** — `sources/themes/cream/light.json` / `dark.json` を §3.2–3.3 の値で作成（`:root` 既定にするなら `light.json` に `"$default": true`）。`npm run build` で Web/Swift 生成物を確認。
2. **既定アイコン `rounded` を埋める** — §4.2 の規約で代表アイコンを `sources/icons/rounded/<category>/` に追加。`sharp` 系は §4.3 に沿って必要分を併設。
3. **`accent.strong` トークンの是非** — §3.4 の用途別アクセントを足すか決定（足すなら全テーマ・全外観に追加）。
4. **パステル多色パレットの置き場** — セマンティックトークン外の「彩りパレット」をどう配布するか（CSS 変数群 / Swift 配列）を設計。
5. このプロファイルは固定ではない。好みが動いたら診断ツールを再実行し、本書を更新する。

> アイコンのスタイル軸（`sources/icons/<style>/<category>/<name>.svg`）と `$default` テーマ指定は **実装済み**（`build/*.mjs`・`CLAUDE.md` 反映済み）。
