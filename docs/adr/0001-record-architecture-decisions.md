# ADR 0001: アーキテクチャ意思決定を ADR で記録する

## Status

Accepted — 2026-06-28

## Context

このリポジトリは single source of truth（`sources/`）からマルチプラットフォーム（Web / Swift）の資産を生成するデザインシステムである。`CLAUDE.md` が *仕組み*、`DESIGN.md` が *審美眼* を定義するが、「なぜその構造・規約・スタイル方針にしたか」という意思決定の経緯はコードにも両ドキュメントにも残らない。スタイル軸の追加、命名規約の変更、アイコン描画署名の変更など、後から「なぜ？」が問われる決定が増えてきた。

## Decision

重要なアーキテクチャ/デザインシステム上の意思決定を、`docs/adr/` 配下に Nygard 形式の ADR として記録する。1ファイル1決定・連番ゼロ埋め4桁・不変（変更時は新 ADR で supersede）。索引は `docs/adr/README.md` に維持する。

## Consequences

- 利点: 決定の背景と帰結が追える。決定の蒸し返し・盲目的な踏襲/破壊を防ぐ。`DESIGN.md`/`CLAUDE.md` は「現在の規約」を、ADR は「なぜ/いつ変えたか」を担う。
- 欠点: 重要な決定のたびに起票の手間がかかる。
- 中立: 些末・整形レベルの決定は対象外（量より質）。起票判断は `/adr` スキルの重要性ゲートに従う。
